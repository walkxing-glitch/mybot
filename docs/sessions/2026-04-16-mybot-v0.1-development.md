### 3. 设计规格

**写入：** `docs/superpowers/specs/2026-04-16-mybot-agent-design.md`（389 行）

**核心架构：**

```
Gateway (CLI + Telegram)
    ↓
Agent Core (LiteLLM + Tool Router + Conversation Manager)
    ↓
┌─────────────────┬──────────────────┐
│   Tools (8个)   │  Memory Engine   │
│  shell/code/    │  Profile演化     │

│  neural_twin/   │  Temporal推理    │

└─────────────────┴──────────────────┘
```

**记忆引擎三大特性（核心差异化）：**

1. **用户画像演化** — 5 个维度（behavior/interest/decision_style/social/focus），每次对话 LLM 输出 diff（update/strengthen/weaken/new_insight/trend_change）
2. **遗忘衰减** — salience = base_importance × e^(-λt) × access_boost × relevance；按类型不同半衰期（episode 30d / fact 180d / preference 365d）
3. **跨会话时间推理** — 记忆带时间戳和 temporal_context，检索时注入时间上下文让 LLM 感知"什么过时了"

## 关键设计决策

| 决策 | 原因 |
|------|------|
| LiteLLM 统一多模型 | 避免为每个 provider 写 adapter |
| 插件化工具注册 | `tools = [...]` 导出 + auto_discover |
| SQLite + FTS5 | 轻量，单文件，容器友好，FTS5 做全文搜索 |
| 记忆按类型分半衰期 | 事实和偏好衰减慢，事件和观察衰减快 |
| 本体论通过 HTTP 调用 | 不复制数据，不耦合，项目独立 |
| launchd 而非 Docker | 纯个人 daemon，Docker Desktop 开机不自启 |
| DI 而非自动注入 MemoryTool | 工具需要 engine 实例，统一在 load_enabled_tools 处理 |

## 9. Neural-Twin 价值验证（同日增补）

**脚本：** `scripts/compare_neural_twin_value.py` — 同一问题、同模型、同记忆库，两个 Agent 实例仅差一个工具。

**问题：** "我最近看中一款 3000 元的蓝牙耳机，想买，但犹豫。你觉得我该不该买？"

### A 组（不带 neural_twin）
LLM 只能给**通用财务建议框架**：要你先回答月收入、备用金、使用时长、型号评测、促销时机……本质是一份模板，问了你 20 个问题才敢下结论。

### B 组（带 neural_twin）
LLM 主动调 `predict_decision` 工具，拿回 4 个关键信号并据此回答：

| 信号 | 数值 | 用途 |
|------|------|------|
| decision score | 0.47 / verdict=think | "这不是该冲动买的事，需要思考" |
| 记忆相似度 | 0.96 想起 "李宏群 2014-04" | 召回有噪声，但被 LLM 合理忽略 |
| 价格预期 | 预测 9132 元 vs 实际 3000 元（-1.07σ） | "从你的消费画像看，这个价格相对安全" |
| 最近月度交易 | 26 笔 | 你是活跃消费者，不是月光 |

### 结论

**价值是真实的、可感知的。** B 组从"泛泛而谈"跳到"基于你自己消费画像的具体判断"，对"该不该买"这种日常决策给出了一个可操作的数据背书。**弱点：** 记忆召回有噪声（人名和金额误关联），但 LLM 把握住了有价值的信号（-1.07σ）。

**下一步值得做的改进：**
- Neural-Twin 数据只到 2025，补 2026 数据后 `/forecast/today` 也能用
- decision network 的语义召回可以继续训练加强
- 可以把"耳机/数码"类目单独做 estimator，减少和人名的交叉

## 12. Ontology 工具扩展：认知层整合视图（同日增补）

### 用户原则

「保持只调用不修改」—— mybot ↔ myontology 永远只是 HTTP 调用，myontology 那边一行代码不动，不启动新的后端服务（比如 backend/myontology/main.py 不启）。

### 发现

之前误判"ontology-engine 里没有聊天记录"。实际上 `/api/situation/{oid}/{YYYY-MM}` 端点**已经把 places + people + wechat 会话 + financial + events + photos + calendar 按月聚合好**，每月只 4KB，LLM 一次吃得下。

### 新增 6 个 operation（全部只是 GET 调用）

| operation | 路径 | size | 延迟 | 场景 |
|-----------|------|-------|-----|------|
| `list_months` | `/api/situation/{oid}/months` | 19KB | 0.2s | 月份索引（哪些月有数据）|
| `get_month(month)` | `/api/situation/{oid}/{month}` | **4KB** | 0.4s | **一月综合视图，金矿级** ⭐ |
| `get_month_narrative(month)` | `.../{month}/narrative` | 0.5KB | 0.4s | 单月自然语言叙述 |
| `get_narrative` | `/api/narrative/{oid}` | 47KB | 0.4s | 整体人生章节叙事 |
| `get_habits` | `/api/habits/{oid}` | 41KB | 即时 | 习惯模式分析 |
| `get_geography` | `/api/lebenswelt/{oid}/geography` | 101KB | 0.5s | 地点×消费×居住×出行融合（按需）|

### 被跳过的端点

- `lebenswelt/temporal` —— 545KB 太大
- `lebenswelt` (root) —— 55KB 图数据（nodes/edges）对 LLM 不友好
- `timeline` —— HTTP 422 缺隐式参数

## 13. 三服务全部 launchd 常驻

**起点：** 第 12 节结尾埋的伏笔 —— ontology-engine 当时还是前台 python，重启即挂。用户发"做"。

### 操作

1. 杀掉前台 python uvicorn 进程（PID 75925）
2. 创建日志目录 `myontology/ontology-engine/logs/`
3. 写 `~/Library/LaunchAgents/com.xingzq.ontology-engine.plist`
   - 系统 python 3.12（无 venv）：`/Library/Frameworks/Python.framework/Versions/3.12/bin/python3`
   - Command：`-m uvicorn server.main:app --port 8003`
   - WorkingDirectory：`myontology/ontology-engine`
   - `KeepAlive=true` + `ThrottleInterval=30` + `ProcessType=Background`

## 14. 记忆污染事件 & 治理（"铁锈" bug）

**症状**：基础设施全部修好后，Telegram 里问"我在北京花了多少钱"仍返回"本体论服务不可用... 没有找到消费记录"。

**诊断**：
- 本地脚本同问题 → 42 秒内 agent 调 `get_geography` 返回 69 万元详细分解
- prod mybot venv → `:8003/api/objects/stats` 返回 200
- 三个 launchd 服务 PID 正常

那为什么 Telegram 还错？检查 `data/memory.db`：

```sql
SELECT created_at, content FROM memories ORDER BY created_at DESC;
```

前 8 条全是 09:21 / 09:28 时段（修复前）写入的**失败叙述**：
- "本体论知识图谱服务（端口8003）当前不可用，影响了地理消费数据的查询"
- "系统未能找到相关记录"
- "用户的记忆系统中没有存储与'北京 消费 花费 金额'相关的信息"（←自指！）

**根因**：memory engine 的自动事实抽取**把工具失败的叙述当成用户事实写进长期记忆**。修复代码之后，语义召回依然把这些"事实"注入 prompt → LLM 复读过时错误。

**治理**：
1. 备份 `data/memory.db` → `data/memory.db.bak-20260416-172942`
2. 删除 11 条污染记忆 + 4 条失败会话摘要（保留合法历史如社区技能讨论、26 笔交易事实）
3. FTS 通过 `memories_fts_delete` 触发器自动同步

**教训（应在产品层防）**：
- **工具失败不应该被抽取成 fact-like memory**。记忆抽取器应对"X 服务不可用"、"未找到 Y"这类**暂态失败叙述**加黑名单或加"transient"标签在召回时过滤。
- **memory 容易"生锈"**：修完代码不够，还要定期检查长期存储里有没有残留错误"事实"。
- **语义召回是把双刃剑**：好的时候让 agent 有连贯性，坏的时候让 agent 困在过去的错误里。

建议 v0.2 加：
- 记忆抽取器的"失败过滤器"（排除含"不可用/失败/未找到/暂停"等关键词的叙述，或对这类仅标记成 TTL 短的临时记忆）
- `mybot memory doctor` CLI 子命令，定期扫污染记忆
